## 1. Executive Summary
Secret materialization ([Book 06 §Ch06](../spec/book-06-runtimes/06-secret-materialization.md)) is specified as a **per-instruction, live-from-the-Broker** operation: at each instruction with a `SecretUse` effect, the runtime resolves the `SecretRef` binding site ([Book 04 §Ch04 §5](../spec/book-04-aos-ir/04-low-ir.md)) to a value, uses it, and discards it. Two properties are left unspecified, and each is load-bearing for a flagship invariant:

1. **Intra-execution stability.** Nothing guarantees that the *same* `SecretRef`, materialized at two instructions **within one Execution**, yields the *same* value. Rotation ([Book 11 §Ch04 §7](../spec/book-11-security/04-secret-broker.md)) changes a value while keeping the reference stable — its comfort claim, "consumers are unaffected because they hold references, not values," is false for a consumer that materializes the same reference twice and needs value-consistency (a session token, a multi-part upload credential, a sign-then-verify pair).
2. **Determinism/replay under secrets.** The now-normative determinism guarantee ([Book 01 §Ch05 §1](../spec/book-01-foundations/05-determinism-and-determinization.md), [Book 06 §Ch03 §1](../spec/book-06-runtimes/03-execution-semantics.md), reflected from RFC-0002) requires *"the same resolved bindings"* for reproducible observable effects. A materialized secret is a **resolved binding whose value is (a) not in the reasoning ledger — P7 forbids it — and (b) not stable under rotation.** *Reconstruction* replay ([Book 06 §Ch03 §1](../spec/book-06-runtimes/03-execution-semantics.md)) injects recorded effect *outcomes* and performs no external effect, so it is unaffected; but *re-execution* replay re-fires effects live, re-materializing each secret from the Broker — across a rotation boundary it materializes a *different* value and performs a secret-dependent effect the recorded run never performed, with **no mechanism to detect the change**. The guarantee names "resolved bindings" as a required-identical input, yet a secret value is a non-recorded input it silently omits.

This RFC (a) makes **intra-execution materialization stability** normative, (b) states the **secret carve-out** to the determinism guarantee precisely, (c) defines replay behavior for secret bindings without ever recording a value, by introducing a secret-free **rotation generation** identifier so a value change is *detectable and fail-closed* rather than silent, and (d) defines the mid-execution **revocation** terminal — memoizing the *value* while re-checking *authority* at every use, so stability never shields a revoked grant — closing the hole beneath "same resolved bindings."

## 2. Problem Statement
Determinism is the mission ([Book 01 §Ch05](../spec/book-01-foundations/05-determinism-and-determinization.md)); secret-freedom (P7) is the security spine ([Book 11 §Ch04](../spec/book-11-security/04-secret-broker.md)). These two invariants intersect at exactly one point — secret materialization — and the intersection is unspecified.

- **Execution reproducibility is stated over "resolved bindings."** [Book 06 §Ch03 §1](../spec/book-06-runtimes/03-execution-semantics.md): *"given the same RuntimeGraph, the same resolved bindings, and the same inputs, execution produces the same observable effects."* A `SecretRef` is a `BindingSite` "resolved **at execution**" ([Book 04 §Ch04 §5](../spec/book-04-aos-ir/04-low-ir.md)). So "resolved bindings" silently depends on the materialized secret *value* — a thing the spec never guarantees is stable and, by P7, can never record to make "the same" checkable.
- **Rotation is advertised as transparent, and it is not.** [Book 11 §Ch04 §7](../spec/book-11-security/04-secret-broker.md): *"new value, same reference … consumers are unaffected."* True across *separate* executions; false *within* one execution and false across a *record→replay* boundary. The spec offers no notion of "the value in effect for the duration of this Execution."
- **RFC-0002 excluded secrets from its input list — deliberately, but without saying so.** Its ledger [MUST be secret-free](0002-replay-semantics-and-recorded-reasoning.md) (§4.3), so the one mechanism that makes replay reproducible for reasoning/`Time`/`Random` **cannot** cover secrets. RFC-0002 even flags `Network(read)` as an open non-recorded input (its Unresolved questions) but never mentions secrets — the sharper case, because a secret is non-recordable by invariant, not by choice.
- **Mid-execution revocation has only grant-level language.** [Book 11 §Ch03 §5](../spec/book-11-security/03-capability-based-security.md) says a revoked grant yields "no *new* invocation," and "in-flight invocations complete or are cancelled per the Capability's policy." That governs the *grant*; it says nothing about a multi-instruction Execution whose *later* `SecretUse` is denied after an *earlier* one succeeded — which is a partial execution needing an explained terminal ([Book 03 §Ch13 §3](../spec/book-03-kernel/13-failure-modes-and-degradation.md), RFC-0004), not a silent stall.

Cost of doing nothing: the "resolved bindings" clause of the reproducibility guarantee is unmet for the entire class of secret-using plans (i.e. most real work) whenever rotation intersects an execution; rotation — a security best practice the spec mandates — becomes a silent intra-execution correctness hazard; and re-execution replay silently performs secret-dependent effects under whatever credential is current, with no way to detect it diverged from the recorded run.

## 3. Alternatives Considered
- **Do nothing.** Rejected: leaves "same resolved bindings" undefined for secrets, makes rotation a latent correctness bug, and makes re-execution replay's observable-equivalence claim silently value-dependent for any secret-using plan.
- **Record the secret value in the ledger to make replay reproducible.** Rejected outright: violates P7 ([Book 06 §Ch06 §3](../spec/book-06-runtimes/06-secret-materialization.md), [Book 11 §Ch04](../spec/book-11-security/04-secret-broker.md)) — the ledger and all IR-adjacent data are secret-free by invariant. A value must never be persisted, and a ledger is durable.
- **Freeze rotation for the duration of any in-flight execution (global rotation lock).** Rejected: rotation exists to shrink exposure windows and to respond to compromise; a long-running execution must not be able to pin a possibly-compromised credential indefinitely, and cross-execution coordination of the Broker's rotation schedule is a scalability and availability hazard.
- **Per-execution materialization memoization + a secret-free generation stamp (chosen).** The runtime materializes each distinct `SecretRef` **at most once per Execution** and reuses that value for the Execution's duration (stability without persistence — the value stays ephemeral, held only in the isolated context). Replay records only the **rotation generation** (an opaque, secret-free version id), so "the secret changed since record" is *decidable and fail-closed* without ever storing the value. Mirrors RFC-0002's "ledger miss ⇒ fail-closed" philosophy: a generation mismatch is an explained refusal, never a silent divergence.

## 4. Proposed Design
**4.1 Intra-execution materialization stability (normative).** For a single `Execution`, a given `SecretRef` **MUST** resolve to a single, stable value for the Execution's lifetime:

> The first `SecretUse` of a `SecretRef` within an Execution materializes its value under the protocol of [Book 06 §Ch06 §2](../spec/book-06-runtimes/06-secret-materialization.md). Every subsequent `SecretUse` of the **same** `SecretRef` **under the same authorizing grant** in the **same** Execution **MUST** observe the **same** value. The memoization is keyed by `(SecretRef, grantedBy)` — the grant is part of the authorization, not incidental ([Book 06 §Ch06 §2](../spec/book-06-runtimes/06-secret-materialization.md) step 2), so a materialization under a *different* grant is a distinct authorization context and **MUST NOT** reuse the memoized value. In the ordinary case of one grant per reference per Execution this is exactly per-`SecretRef` stability. The value remains ephemeral (held only in the runtime's isolated context, never persisted, never cached across executions — [Book 06 §Ch06 §5](../spec/book-06-runtimes/06-secret-materialization.md)); stability is a within-execution memoization, not storage.

**The memoization caches the value, never the authorization.** Every `SecretUse` — including those served from the memoized value — **MUST** re-run the capability (P8) and policy/approval (P9) checks of [Book 06 §Ch06 §2](../spec/book-06-runtimes/06-secret-materialization.md) steps 2–3. Only step 4's resolution is elided by the memoization. This is what keeps §4.1 compatible with revocation, which is normatively *"prompt and complete: after revocation, subsequent invocations under it are denied, **even past any caching**. A just-revoked grant cannot be used for a new effect"* ([Book 11 §Ch03 §5](../spec/book-11-security/03-capability-based-security.md)) — a rule that speaks directly to a cache like this one.

**Rotation and revocation are therefore not symmetric**, and must not be conflated:
- **Rotation** (same reference, new value, authority unchanged — [Book 11 §Ch04 §7](../spec/book-11-security/04-secret-broker.md)) takes effect at **Execution boundaries**. A rotation landing mid-flight does not change a value already materialized in that Execution; it applies to the *next* Execution that materializes the reference. This is what makes "consumers are unaffected" true as stated, by giving "consumer" the natural boundary of one Execution.
- **Revocation** (authority withdrawn) takes effect **immediately, mid-Execution**, per [Book 11 §Ch03 §5](../spec/book-11-security/03-capability-based-security.md). The memoized value confers no authority: the next `SecretUse` under a revoked grant fails its capability check and the Execution enters §4.5. Value-stability **MUST NOT** shield a revoked grant.

**4.2 The secret carve-out to the determinism guarantee (normative).** The determinism guarantee ([Book 01 §Ch05 §1](../spec/book-01-foundations/05-determinism-and-determinization.md), [Book 06 §Ch03 §1](../spec/book-06-runtimes/03-execution-semantics.md), reflected from [RFC-0002 §4.1](0002-replay-semantics-and-recorded-reasoning.md)) is amended to name secrets explicitly:

> Given identical Low IR, identical resolved bindings **(including identical materialized secret values, guaranteed within an Execution by §4.1 and identified across executions by rotation generation, §4.4)**, identical declared inputs, and identical recorded outputs for every `CapturedReasoning` node and every `Time`/`Random` effect, execution produces identical observable effects and outputs.

The secret value is thus acknowledged as an input to observable behavior that is **secret-free-by-necessity unrecordable**: the guarantee holds *modulo* the materialized secret values, which the platform pins **within** an Execution (§4.1) but does **not** carry **across** a record→re-execution boundary. Re-execution replay reproduces a secret-dependent observable effect **only when the referenced secret is at the same rotation generation as at record time** (§4.3); reconstruction is unaffected, injecting recorded outcomes.

**4.3 Replay with secret bindings (normative).** Replay-with-record has two variants ([Book 06 §Ch03 §1](../spec/book-06-runtimes/03-execution-semantics.md)); secrets are treated per variant:
- **Reconstruction** performs no external effect and injects every effectful instruction's recorded outcome. It therefore materializes **no** secret and re-fires no secret-dependent effect — it is a pure function of `(Low IR, bindings, inputs, record)` and is unaffected by rotation. This RFC requires only that reconstruction **MUST NOT** materialize a secret, consistent with its no-external-effect contract.
- **Re-execution** re-fires effects live, re-materializing each secret from the Broker under the full protocol ([Book 06 §Ch06 §2](../spec/book-06-runtimes/06-secret-materialization.md)) with capability (P8) and policy/approval (P9) checks against **live** state — its guarantee is already *identical observable behavior up to fail-closed policy denial* ([Book 06 §Ch03 §1](../spec/book-06-runtimes/03-execution-semantics.md)). This RFC extends that: before re-firing a secret-dependent effect, the runtime **MUST** compare the reference's **current rotation generation** (§4.4) to the generation recorded at the original run. On a match it proceeds; on a mismatch (rotated or revoked since record) it **MUST fail closed** with an explained `SecretGenerationChanged` condition — it **MUST NOT** re-perform a secret-dependent effect under a credential the recorded run never used, nor silently report the divergence as a determinism failure.

**4.4 Rotation generation (normative, secret-free).** The `Secret` Resource ([Book 02 §Ch07 §5](../spec/book-02-resource-model/07-core-resource-catalog.md), [Book 11 §Ch04 §7](../spec/book-11-security/04-secret-broker.md)) gains a monotonic, opaque **`rotationGeneration`** — a version counter incremented on every rotation/revocation, carrying **no** value material (it is an integer/opaque token, not a hash of the value, so it leaks nothing under P7). Materialization audit ([Book 11 §Ch04 §6](../spec/book-11-security/04-secret-broker.md), [Book 11 §Ch09](../spec/book-11-security/09-audit-and-attribution.md)) records, by reference, *which generation* was materialized for execution E under grant G — strengthening P10 attribution ("which secret, which version, was used where") while preserving P7 ("never what it was"). The fresh-run generation stamp is the sole datum replay needs to make §4.3's comparison; it lives in audit/Execution status, never in the runtime and never as a value.

**4.5 Mid-execution revocation terminal (normative).** Revocation is prompt and complete ([Book 11 §Ch03 §5](../spec/book-11-security/03-capability-based-security.md)), so it is handled identically **whether or not** the reference has already been materialized in the Execution — the memoization (§4.1) changes the *value* path, never the *authority* path:
- **Before the first `SecretUse`.** Materialization fails its capability check and fails closed ([Book 06 §Ch06 §2](../spec/book-06-runtimes/06-secret-materialization.md)).
- **After the first `SecretUse`** (a value is memoized). The next `SecretUse` re-runs the capability check (§4.1) and is **denied** — the memoized value is not used, and no *new effect* proceeds under the revoked grant, exactly as [Book 11 §Ch03 §5](../spec/book-11-security/03-capability-based-security.md) requires "even past any caching."

In both cases the Execution reaches an **explained terminal** ([Book 03 §Ch13 §3](../spec/book-03-kernel/13-failure-modes-and-degradation.md)): a partial Execution whose completed effects are compensated where declared, and — where a compensating effect itself needs the now-revoked secret and so cannot run — the `CompensationFailed` condition and `RemediationTask` escalation of [RFC-0004](0004-compensation-failure-terminal-and-escalation.md). No secret-dependent effect proceeds with a missing, revoked, or stale credential (P7/P8 fail-closed); no partial state is left silent (the "no silent partial success" guarantee, [Book 03 §Ch13 §3](../spec/book-03-kernel/13-failure-modes-and-degradation.md)). An in-flight *invocation* already underway completes or is cancelled per the Capability's policy ([Book 11 §Ch03 §5](../spec/book-11-security/03-capability-based-security.md)); this RFC governs only whether a *subsequent* `SecretUse` may proceed — it may not.

## 5. Tradeoffs
**Gain:** "same resolved bindings" becomes true for secrets; rotation stops being a silent intra-execution correctness hazard; re-execution replay gains fail-closed detection of a secret change instead of silently diverging (reconstruction is unaffected by construction); P10 attribution gains secret *version* granularity at zero P7 cost.
**Give up:** the runtime must memoize secret values per-Execution (a small, ephemeral in-context map — no new persistence); the Broker/`Secret` Resource must maintain and audit a rotation generation; replay must carry and compare the recorded generation. A very long-running Execution pins a secret's *value* for its duration, marginally widening that value's use window against **rotation** (bounded by execution length). It does **not** widen it against **revocation**: because §4.1 re-checks authorization at every `SecretUse`, revoking the grant cuts the Execution off immediately (§4.5), so a credential believed compromised is contained at once rather than at the next Execution boundary.

## 6. API Changes
Additive; no signature changes:
- Secret Broker materialization ([Book 06 §Ch06 §2](../spec/book-06-runtimes/06-secret-materialization.md)) returns, alongside the injected value, the current `rotationGeneration` for audit/record — the *value* injection path is unchanged and still exposes no value up any API.
- Runtime interface ([Book 06 §Ch02](../spec/book-06-runtimes/02-runtime-interface.md)): the per-Execution memoization of §4.1 is a runtime-internal obligation; `describe()` declares whether the runtime enforces intra-execution secret stability (required for replay-mode conformance). In replay mode, the recorded generation stamp rides the existing `executionCtx` beside the reasoning ledger ([RFC-0002 §6](0002-replay-semantics-and-recorded-reasoning.md)).
- Kernel API: `Secret` status gains `rotationGeneration`; `Execution` status records the generation materialized per `SecretRef` (by reference, secret-free) — additive (P6).

## 7. Resource Changes
- `Secret` ([Book 02 §Ch07 §5](../spec/book-02-resource-model/07-core-resource-catalog.md)): additive `status.rotationGeneration` (opaque, monotonic, value-free).
- `Execution` ([Book 02 §Ch07](../spec/book-02-resource-model/07-core-resource-catalog.md)): additive `status.secretBindings[]` — for each materialized `SecretRef`, the reference and the generation materialized (never a value). Secret-free, retained with the Execution, and part of the same attribution surface as the reasoning ledger ([RFC-0002 §7](0002-replay-semantics-and-recorded-reasoning.md)). No lifecycle change; Execution stays one-shot/terminal.

## 8. Event Changes
Additive: `secret.materialized` gains the `rotationGeneration` field (by reference, secret-free — extends the existing materialization audit Event, [Book 11 §Ch04 §6](../spec/book-11-security/04-secret-broker.md), [Book 14 §Ch01](../spec/book-14-observability-governance/01-event-model.md)); a `secret.rotated` domain Event carries the new generation (no value). A replay generation mismatch emits `execution.replayRefused` with the `SecretGenerationChanged` reason. Per-subject ordering on the `Secret`/`Execution` subjects suffices ([Book 03 §Ch03 §5](../spec/book-03-kernel/03-event-bus.md)).

## 9. Security Impact
This RFC *strengthens* P7 and P10 and introduces **no** new secret flow. The rotation generation is an opaque version counter with no value material — it cannot be inverted to a value, so it is safe in Events, audit, and `Execution` status exactly where a `SecretRef` already is. The per-Execution memoization keeps the value ephemeral and in-context (no persistence, no cross-execution cache — [Book 06 §Ch06 §5](../spec/book-06-runtimes/06-secret-materialization.md) unchanged). §4.3 explicitly forbids re-execution replay from re-performing a secret-dependent effect under a credential the recorded run never used, closing a would-be "replay uses a newer/other secret" path. Security review required (touches P7, P8, P9, and the P1 determinism invariant reflected from RFC-0002). Critically, the memoization is a **value** cache and never an **authority** cache (§4.1): every `SecretUse` re-runs the capability and policy checks, so revocation stays *"prompt and complete … even past any caching"* ([Book 11 §Ch03 §5](../spec/book-11-security/03-capability-based-security.md)) — a §4.1 that elided the check would have silently defeated prompt revocation, which is why the split is normative and directly tested (§11). SI-checklist ([Book 11 §Ch11](../spec/book-11-security/11-security-invariants.md)) gains an invariant: *a secret value materialized within an Execution is stable for that Execution and its authorization is re-checked at every use; a revoked grant is denied even where a value is memoized; a value change across replay is fail-closed, never silent.*

## 10. Performance Impact
Per-Execution memoization removes redundant Broker round-trips for repeated `SecretUse` of one reference — a net reduction in materialization calls on the hot path. The generation stamp is one small field per materialization. Replay's generation comparison is O(1) per reference. No durable-write growth beyond the small `secretBindings[]` list (bounded by distinct references per Execution).

## 11. Testing Strategy
- Conformance: a runtime **MUST** pass an *intra-execution stability* test — a plan that materializes one `SecretRef` at two instructions, with a rotation injected between them, **MUST** observe one stable value.
- Conformance: re-execution replay with matching generation ⇒ proceeds; with a changed generation ⇒ fail-closed `SecretGenerationChanged`, never a divergent execution or a fresh secret-dependent effect; reconstruction replay materializes no secret at all.
- Failure-injection: grant revoked **before** first `SecretUse` ⇒ materialization denied, explained terminal + compensation (RFC-0004 escalation where compensation itself needs the revoked secret).
- **Revocation beats the memoization** (the §4.1 value/authority split): grant revoked **after** a value is memoized ⇒ the next `SecretUse` is **denied**, the memoized value is not used, and no new effect fires — asserting [Book 11 §Ch03 §5](../spec/book-11-security/03-capability-based-security.md)'s "even past any caching" against precisely the cache §4.1 introduces.
- Property: for any secret-using plan, two fresh runs at the same generation with identical other inputs produce identical observable effects; across a rotation, re-execution replay refuses rather than diverges.

## 13. Migration Strategy
Additive and pre-1.0; no stored artifact changes meaning. `rotationGeneration` defaults to an initial generation for existing `Secret`s; runtimes without replay mode need only honor §4.1 (a local memoization) to stay conformant for fresh execution; replay-mode conformance requires the generation comparison at the milestone RFC-0002 names for replay.

## 14. Risks
- **Long-execution value pinning.** §4.1 pins a *value* for an Execution's duration, marginally widening exposure to a **rotated-away** value — mitigated by execution-length bounds. It does not widen exposure to a **revoked** grant: §4.1 re-checks authorization at every `SecretUse`, so revocation contains a suspected compromise immediately (§4.5), satisfying [Book 11 §Ch03 §5](../spec/book-11-security/03-capability-based-security.md)'s "even past any caching."
- **Reading §4.1 as an authorization cache.** The central misreading risk: a memoization that elided the capability check would silently defeat prompt revocation. §4.1 states the value/authority split normatively and §11 tests it directly, but the reflecting PR must keep the two paths visibly distinct in Book 06 §Ch06's prose.
- **Generation-as-side-channel.** A monotonic counter could, in principle, signal rotation cadence — accepted: cadence is already observable via `secret.rotated`/audit, and the counter carries no value material.
- **Interaction with RFC-0002 recovery/partial-failure.** Kernel-restart recovery ([Book 03 §Ch13 §4](../spec/book-03-kernel/13-failure-modes-and-degradation.md)) mid-Execution re-materializes; §4.1 must survive recovery (the memoization is per-Execution *logical* identity, re-established on resume against the recorded generation) — flagged for joint review with the RFC-0002 recovery path.

## 15. Future Improvements
A "sealed replay" mode that, together with RFC-0002's potential `Network(read)` ledgering, snapshots a fully reproducible execution against a pinned generation set for forensic replay; provider-native generation identifiers (KMS key version, OAuth token id) mapped onto `rotationGeneration` so external rotation is observed without polling.

### Resolved questions
- **Per-`SecretRef` or per-(`SecretRef`, grant) stability?** Per-(`SecretRef`, **grant**), per §4.1. The authorizing grant is part of the materialization protocol — holding the reference is *"necessary but **not sufficient**"* ([Book 06 §Ch06 §2](../spec/book-06-runtimes/06-secret-materialization.md) step 2) — so two materializations under different grants are distinct authorization contexts and **MUST NOT** silently share a memoized value. In the normal case (one grant per reference per Execution) this collapses to per-`SecretRef`, so the stricter key costs nothing and is safe by default.
- **Is a generation change an automatic drift signal ([RFC-0003](0003-drift-detection-via-shadow-sampling.md))?** No — **orthogonal**. Drift is divergence between a determinized `Capability`'s output and fresh reasoning over the same typed inputs; a rotation changes an *effect's* credential, not the reasoning's input→output relation. Treating rotation as drift would retire healthy Capabilities on routine rotation — creating pressure against a security best practice the spec mandates ([Book 11 §Ch04 §7](../spec/book-11-security/04-secret-broker.md)). The generation is recorded in the evidence (§4.4) so a correlation can be *investigated*, but it never auto-retires.
- **Expose `rotationGeneration` to policy ([Book 11 §Ch06](../spec/book-11-security/06-policy-engine.md))?** Yes (§4.4). It is an opaque, secret-free token already carried in audit, so exposing it adds no P7 surface, and it enables a real incident-response lever: policy may require "materialize only the latest generation" for high-consequence classes, forcing fresh credentials after a suspected compromise.
