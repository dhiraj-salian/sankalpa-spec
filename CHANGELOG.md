# Changelog

All notable changes to the Sankalpa specification are recorded here. The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/); this repository versions the **specification**, following [Semantic Versioning](https://semver.org/) applied to normative meaning (see [`process/versioning-and-stability.md`](process/versioning-and-stability.md)).

## [Unreleased]

### Changed
- **RFC-0007 reflected into `spec/` and marked `Final`** (Phase 2 hardening batch 2). Normative changes:
  - **Scheduling admission liveness (RFC-0007)**: the Scheduler was specified to "apply fairness across workspaces/tenants", "honor per-work **deadlines** and **priorities** without **starving** low-priority tenants", and "hold or **shed** work deterministically" — all **asserted with no mechanism, no fields, and no lifecycle outcome**. No schedulable Resource carried a priority or deadline, so the promise named inputs that did not exist; priority without aging or reservation *cannot* deliver starvation-freedom; and `Pending` ("cannot yet begin") had no exit, so an unadmittable `Execution` could sit **indefinitely and invisibly** — the one silent stall in a spec that forbids them everywhere else. Adds **`schedulingClass`** (policy-mapped, capability-gated so priority is governed, not self-asserted) and optional **`deadline`** as `spec` fields; a normative **starvation-freedom floor** (bounded wait via aging and/or *static* per-tenant weighted-fair reservation — learned reservations are excluded because fairness is a guarantee, not a choice among already-equivalent options per Book 10 §05); an explicit **hold vs. shed** split where a held unit always carries a surfaced `AdmissionPending` condition and an unadmittable one reaches `Failed`/**`Shed`**; and a **`DeadlineExceeded`** terminal that compensates and escalates via `CompensationFailed` where compensation itself fails. Disambiguates the two senses of "admission" (store- vs run-), so `Pending` is bounded and never silent. A rescheduled Execution carries its original deadline. Touches Books 02 §04/§07, 03 §07/§13, 14 §02.
  - New Glossary terms: aging, DeadlineExceeded, run-admission, scheduling class, shed (work).
- **RFC-0005 reflected into `spec/` and marked `Final`** (Phase 2 hardening batch 2). Normative changes:
  - **Secret materialization stability and the secret carve-out (RFC-0005)**: materialization was specified per-instruction and live-from-the-Broker, with nothing guaranteeing that the *same* `SecretRef` materialized twice **within one Execution** yields the same value — so a rotation landing mid-run could silently change a resolved binding (breaking a session token, a multi-part upload, a sign-then-verify pair), leaving the determinism guarantee's "same resolved bindings" clause unmet for any plan that uses one credential twice. Book 11 §04's "consumers are unaffected because they hold references, not values" was true only across *separate* executions. Adds **intra-execution stability** keyed `(SecretRef, grantedBy)`; makes explicit that the memoization caches the **value, never the authority** — every `SecretUse` re-runs the capability and policy checks, so revocation still bites *"even past any caching"* (Book 11 §03 §5), which now has a named cache to speak to. Rotation and revocation are **no longer symmetric**: rotation takes effect at Execution boundaries, revocation immediately. Names the **secret carve-out** to the determinism guarantee (a materialized secret is a resolved binding P7 makes unrecordable) and adds the secret-free **`rotationGeneration`**, letting re-execution replay fail closed on a changed credential rather than silently performing an effect the recorded run never performed; reconstruction replay is unaffected, materializing no secret at all. Touches Books 01 §05, 02 §07, 04 §04, 06 §03/§06, 11 §03/§04.
  - New Glossary terms: intra-execution secret stability, rotation generation, secret carve-out (determinism).
- **RFC-0009 reflected into `spec/` and marked `Final`** (Phase 2 hardening batch 2). Normative changes:
  - **Channel identity binding and assurance (RFC-0009)**: a channel's transport identity was *"mapped to a Session/User"* — a **mapping, never an authentication** — while the same chapters declare the channel **untrusted**. Front-door auth (tokens/mTLS/OIDC) does not reach a Telegram message or an email `From`, so the Kernel took an untrusted adapter at its word about *who is speaking*: a spoofed or compromised handle could issue Intents and continue Conversations as the user, and cross-channel continuity ("starts in Slack, continues by email") silently extended a Session's authority to an unauthenticated leg. Adds the **`ChannelBinding`** core kind (verified enrollment from an already-authenticated context; workspace-scoped, revocable, audited), a fixed totally-ordered **assurance level** (`low`<`medium`<`high`) that **caps a Session's effective authority**, **step-up** for consequential actions to the Web Runtime (which already hosts approval, so the approval page *is* the step-up), and cross-channel continuity that **re-binds rather than inherits**. Attribution now records assurance, so audit answers not just *who* but *how strongly proven*. Touches Books 02 §07, 11 §02/§07/§08, 13 §02/§03/§06.
  - New Glossary terms: assurance level, channel binding, enrollment, step-up.
- **RFC-0008 reflected into `spec/` and marked `Final`** (Phase 2 hardening batch 2). Normative changes:
  - **Grant re-authorization on upgrade (RFC-0008)**: capability grants were established once, at install, and the upgrade flow had **no re-authorization step** — so an upgrade that kept the same declared capabilities but shipped different code (a compromised key, or a package-ownership takeover) silently inherited `payments`, secret-materialization, and egress grants for code no one reviewed. Grants now record an **authorized-against identity** (version + publisher + signing key), and upgrade re-evaluates them: broadened authority is surfaced for explicit authorization; a **signing-identity change re-authorizes sensitive-class capabilities even when the capability set is unchanged**; same-identity non-broadening upgrades may carry forward frictionlessly; narrowed-away grants are auto-revoked. Rollback to a previously-authorized version needs no new consent, but a target *wider than the current grant set* is treated as an upgrade — closing the ship-narrow-then-roll-back evasion. Book 12 §04's "a compromised publisher cannot escalate beyond the capabilities a workspace explicitly grants" is tightened: true for *net-new* authority, it did not cover **re-use of granted authority by substituted code**. Touches Books 11 §03/§06, 12 §04/§05.
  - New Glossary terms: authorized-against identity, grant carry-forward, upgrade re-authorization.
- **RFC-0006 reflected into `spec/` and marked `Final`** (Phase 2 hardening batch 2). Normative changes:
  - **Base-version three-way Knowledge reconcile (RFC-0006)**: vault⇄graph sync no longer rests on optimistic concurrency alone, which a **direct/offline vault edit cannot satisfy** — such an edit observes no `resourceVersion`, so compare-and-swap had nothing to check and the "no lost update" guarantee was unbacked for the vault's primary edit path. Adds the controller-managed **`syncedFrom`** merge-base stamp to note front-matter and a three-way reconcile (fast-forward / merge / conflict) keyed on `generation`, with a fail-safe that an absent, malformed, or unknown base is conflict-surfaced and never merged. Trust now adjudicates only where the sides differ in trust; same-trust conflicts are decided by the base. Conflicted Knowledge is **withheld from planning retrieval** (indeterminate, not low-trust) rather than down-weighted. Touches Books 09 §03/§05/§06.
  - New Glossary terms: synced-base stamp, three-way Knowledge reconcile.
- **RFCs 0002–0004 reflected into `spec/` and marked `Final`** (Phase 2 hardening set). Normative changes:
  - **Recorded-reasoning carve-out (RFC-0002)**: the determinism guarantee now explicitly includes recorded `CapturedReasoning`/`Time`/`Random` outputs; introduces the secret-free, content-addressed **reasoning ledger** on `Execution` and two replay modes — **reconstruction** (no effects; defines conformance/golden-file tests) and **re-execution** (effects re-fire under live policy). Touches Books 01 §05, 04 §04/§07, 06 §03, 10 §03.
  - **Shadow sampling for drift (RFC-0003)**: replaces the inert "diverging from fresh reasoning" language with a policy-governed, non-blocking, version-bucketed shadow-sampling mechanism; adds `shadowSource` to substituted invocations. Touches Books 10 §06, 05 §06, 04 §04, 01 §05.
  - **Compensation-failure condition (RFC-0004)**: adds the `CompensationFailed` terminal *condition* (not a phase), best-effort continuation, one-level cascade bound, mandatory escalation to a new **`RemediationTask`** core kind (two-party ack for high-consequence residuals), a failure-taxonomy row, and reconciles Book 10's terminal-state vocabulary. Touches Books 06 §03, 03 §13, 02 §04/§07.
  - New Glossary terms: reasoning ledger, fresh/replay-with-record/reconstruction/re-execution; drift, shadow sampling/source, version bucket, re-validation window; compensation failure, remediation task, residual inconsistency.

### Added
- Initial repository scaffold: governance, contribution, and process framework.
- Templates for RFC, ADR, AEP, implementation plan, testing plan, security review, performance review, design doc.
- Specification skeleton (Books 00–15) with per-Book scope and chapter outlines.
- **Framing Books authored to Draft-complete — `spec/` is now fully Draft-complete (all 16 Books)**:
  - **Book 01 (Foundations)**: mission & binding non-goals, core philosophy, the layered architecture with per-layer ownership, determinism & determinization defined precisely, and influences & prior art (the principles chapter P1–P13 was already written).
  - **Book 00 (Preamble)**: how to read the spec, conformance language (RFC 2119 + scoped/versioned conformance), notation, the terminology single-source-of-truth policy, and the relationship to implementation.
  - **Book 15 (Appendices)**: the capstone end-to-end walkthrough (one intent through every Book), the consolidated notation reference, the worked-examples catalog, the concept↔prior-art mapping, the FAQ & anti-patterns, and the glossary pointer.
- **Book 13 (Interfaces & Channels) authored to Draft-complete**: the integrated Web Runtime (host of the security-critical credential and approval pages), the channel model (channels carry, they do not decide), reference-channel mappings, the API gateway (external face of the Kernel API, not a second path), Conversation & Session (continuity that is not the memory anti-pattern), approval & human-in-the-loop rendering, and governed bidirectional MCP integration.
- **Book 12 (Packages & Marketplace) authored to Draft-complete** (specification body of AEP-0003): everything-is-installable and install≠privilege, the package manifest, deterministic dependency resolution, signing & supply-chain provenance, the reconciled install lifecycle with capability grants and rollback, the Marketplace (discovery/trust/moderation that never relaxes install-time guarantees), and compatibility & stability across version spaces.
- **Book 07 (Controllers & Reconciliation) authored to Draft-complete**: the reconciliation model (level-triggered, idempotent, single-writer), the Controller Runtime's guarantees (coalesced triggers, leader election, backoff), the Controller SDK contract, consistency & conflict handling (optimistic concurrency, no cross-Resource transactions), deterministic failure & recovery, and a worked "writing a controller" walkthrough. Completes the core substrate mechanics.
- **Book 14 (Observability & Runtime Governance) authored to Draft-complete**: the complete Event model & taxonomy, metrics (SLIs/SLOs, secret-free labels), end-to-end tracing (IR by content hash), structured logging with the hard secret-freedom guarantee, audit as an observability view, the runtime policy checkpoint (the third P9 enforcement point, fail-closed against live state), Experience integration, and operability. Resolves the pervasive forward references to the Event taxonomy and runtime checkpoint.
- **Book 09 (Knowledge) authored to Draft-complete**: knowledge vs. memory (rejecting the memory-as-context anti-pattern), the knowledge taxonomy with mandatory provenance & trust, the Obsidian-compatible vault model, the graph model (replaceable backend), reconciled bidirectional vault⇄graph synchronization, knowledge in planning (non-secret, trust-weighted context), and knowledge packs. Completes the learning subsystem with Book 10.
- **Book 10 (Experience) authored to Draft-complete**: Experience as a first-class Resource (not memory), the Experience anatomy, the capture pipeline (deriving secret-free Experience from the Event stream), feedback into Knowledge, feedback into planning & cost models, the determinization *discovery* engine (pairing with Book 05's substitution passes to realize P13), and metrics & evaluation (ran ≠ succeeded). Closes the learning loop.
- **Book 11 (Security) authored to Draft-complete**: threat model (reasoning-context leakage as the defining threat), trust boundaries, capability-based security (P8), the Secret Broker and its separate protected store (P7), out-of-band credential acquisition, the Policy Engine and its three checkpoints (P9), the Approval Engine, identity/users/sessions, audit & attribution (P10 without exposing secrets), plugin isolation, and the consolidated SI-1…SI-12 security-invariant checklist.
- **Book 08 (Planners) authored to Draft-complete**: the role of planning as the compiler front-end and sole home of non-determinism, Intent→Goal derivation with human-in-the-loop clarification, the AEP-0001 planner interface, planner isolation & the no-secrets/no-execution-authority disciplines, the non-determinism boundary (typed Reasoning nodes that recede via determinization), reference-planner adapters, and the property-based planner conformance suite. Closes the intent→execution loop.
- **Book 06 (Runtimes) authored to Draft-complete**: runtime abstraction and the compiler/runtime boundary, the AEP-0002 runtime interface, execution semantics (determinism, idempotency/retries, saga compensation), runtime selection (fidelity/policy gates), reference-runtime mappings, secret materialization at execution only, and the runtime conformance suite. Completes the intent→execution arc.
- **Book 05 (The Compiler) authored to Draft-complete**: pipeline overview, pass framework (analysis/transform, purity, semantics preservation), optimization passes, the pre-execution policy-validation pass, the two-stage lowering framework, determinization passes (P13 substitution with safety gates and reversibility), the diagnostic model, and compilation-as-a-Resource.
- **Book 03 (The Kernel) authored to Draft-complete**: microkernel rationale, the Kernel API, the Event Bus, Managers overview, and full chapters for the Resource Manager, Capability Manager, Scheduler & Runtime Manager, Compiler Manager, Plugin & Package Managers, Lifecycle Manager & Controller Runtime, the security/knowledge/experience Manager integration contracts, the Observability Manager, and failure modes & degradation.
- **Book 02 (The Agent Resource Model) authored to Draft-complete**: uniform resource rationale, resource anatomy, desired-vs-actual reconciliation contract, lifecycle model, versioning & schema evolution, relationships & references (incl. secret references), core resource catalog, storage & consistency.
- **Book 04 (AOS IR) authored to Draft-complete**: IR rationale, design principles (IR-P1…P10), High IR, Low IR, type system, effect system, serialization & content addressing, verification, versioning & compatibility, worked examples.
- Seed decisions: ADR-0000, ADR-0001, ADR-0002; RFC-0000, RFC-0001; AEP-0000.
- Phase 0 research index for prior-art and pattern studies.
- Glossary, roadmap, security policy, code of conduct.
